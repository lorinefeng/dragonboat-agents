# DragonBoat v0.1 Data Contracts

DragonBoat v0.1 uses local, append-only data contracts.

The goal is to make each run observable and replayable without binding the core product to any single coding-agent CLI.

## Storage Layout

The canonical v0.1 runtime layout should be:

```text
.dragonboat/
  runs/
    run_<id>/
      events.ndjson
      state.json
      crew.json
      tasks/
        task_<id>/
          task-packet.json
          task-packet.md
      evidence/
        task_<id>/
          evidence-bundle.json
          summary.md
      mailbox/
        inbox/
          agent_<id>.ndjson
        outbox/
          agent_<id>.ndjson
      logs/
        cmd_<id>.log
      artifacts/
```

`events.ndjson` is canonical. Other files are materialized artifacts for agent readability, validation, local web rendering, and replay.

## Local Event

Schema: [`schemas/v0/local-event.schema.json`](../schemas/v0/local-event.schema.json)

Events are append-only JSON objects, one per line.

Each event contains:

- stable schema version
- event id
- run id
- monotonically increasing sequence number
- timestamp
- event type
- actor
- optional task, message, or command reference
- event-specific payload

The command deck should render state from the event stream rather than trusting a mutable UI state file.

### Crew Supervision

Crew supervision records whether live rowers reached the milestones the steerer was waiting for.

Minimum payload fields:

- `agents`: rower ids being supervised.
- `expectations`: requested milestones such as `intent_confirmed`, `status`, and `evidence`.
- `report`: per-agent `met`, `missing`, `blocked`, and `complete` projection.

`watchdog.continuation.recorded` remains a Codex Stop-hook wake-up bridge. `supervision.wait.*` is the live steering-loop record used when Codex intentionally waits for rower progress in the current turn.

The local web command deck can derive crew state, task state, mailbox timelines, agent console output, and evidence panels from these events.

## Task Packet

Schema: [`schemas/v0/task-packet.schema.json`](../schemas/v0/task-packet.schema.json)

A task packet is the handoff from steerer to rower.

The JSON version is canonical. The Markdown version is generated from the JSON packet so CLI agents can read it naturally.

Every task packet includes:

- task id
- run id
- title
- assigned rower
- creating steerer
- objective
- relevant context
- worktree and path boundary
- dependencies
- acceptance criteria
- expected evidence
- reporting requirements

The packet should be strict enough for automation and readable enough for agents.

## Evidence Bundle

Schema: [`schemas/v0/evidence-bundle.schema.json`](../schemas/v0/evidence-bundle.schema.json)

An evidence bundle is the rower's completion package.

It must include:

- task id
- run id
- submitting agent
- success, failure, blocked, or partial status
- summary
- changed files
- commands run
- check results
- mailbox references
- artifacts
- risks
- unresolved questions
- next-step recommendations

The steerer should review evidence bundles before accepting or integrating any work.

## Structured Handoff

`handoff.submitted` is the semantic peer delivery event used when a rower output must be consumed by another agent.

Minimum payload fields:

- `handoffId`
- `from`
- `recipient`
- `taskId`
- `summary`
- `claims`
- `sources`
- `confidence`
- `open_questions`
- `required_action`
- `artifact_path`
- `ack_required`

`handoff.acknowledged` records that the recipient has read, consumed, or questioned the handoff. A sent handoff is not the same as a consumed handoff; evidence that depends on an ack-required handoff should stay rejected until the ack exists.

`task.completed` is the rower-side closure record for a task. The preferred CLI path is `dragonboat task complete`, which writes the structured handoff, evidence submission, evidence-gate result, task status, crew-member status, and completion event together.

## Context Bundle

Schema: [`schemas/v0/context-bundle.schema.json`](../schemas/v0/context-bundle.schema.json)

A context bundle is the provider-neutral runtime handoff used when DragonBoat needs to pass current run state across heterogeneous agent adapters.

It is not a raw transcript copy and it does not replace a task packet. A task packet defines the assignment; a context bundle carries the current coordination state around that assignment.

Every context bundle includes:

- run id and creation time
- recipient identity and platform
- optional task summary
- current crew roster
- relevant mailbox messages
- advisor notes when the recipient is the steerer
- recent events
- evidence summaries
- adapter hints, including preferred format and provider-neutral intent
- constraints that preserve DragonBoat semantics

The CLI can emit bundles as Markdown for agent readability or JSON for adapter tooling:

```sh
.dragonboat/bin/dragonboat context bundle --agent <agentId> --task <taskId>
.dragonboat/bin/dragonboat context bundle --agent <agentId> --task <taskId> --format json
```

## Shared Fact Board And Context Delta

Schemas:

- [`schemas/v0/shared-fact-board.schema.json`](../schemas/v0/shared-fact-board.schema.json)
- [`schemas/v0/context-delta.schema.json`](../schemas/v0/context-delta.schema.json)

The shared fact board is the steerer-facing synthesis surface derived from the local event ledger. It separates:

- confirmed facts
- unverified claims
- conflicting claims
- missing evidence
- pending handoffs
- accepted conclusions

Context delta is the rower-facing incremental sync payload. It carries only the new facts, conflicts, pending handoffs, open questions, and artifact paths since a known event sequence.

```sh
.dragonboat/bin/dragonboat fact board --latest --format markdown
.dragonboat/bin/dragonboat context delta --to <agentId> --since <seq> --latest --format json
```

## Delegation Economics

Schema: [`schemas/v0/delegation-fit.schema.json`](../schemas/v0/delegation-fit.schema.json)

Delegation Economics v0 decides whether a task should become a crew task before the steerer launches rowers.

It records:

- seven `0..3` score fields
- hard blockers that force `single_agent_default`
- computed `fit_score`
- decision: `crew_strong_fit`, `crew_candidate`, or `single_agent_default`

The CLI can emit a Markdown or JSON assessment:

```sh
.dragonboat/bin/dragonboat delegate assess --context-amortization 3 --parallel-split 3 --interface-stability 3 --acceptance-executability 3 --low-cost-rower-fit 3 --shared-state-penalty 0 --runtime-drift-penalty 1
```

## Evidence Gate

Schema: [`schemas/v0/evidence-gate.schema.json`](../schemas/v0/evidence-gate.schema.json)

`evidence.submitted` is a claim. The evidence gate decides whether that claim is reviewable.

Minimum checks:

- required durable mailbox or structured handoff before evidence
- recipient ack before evidence review when the handoff is ack-required
- acceptance proof before a done claim
- tracked workspace visibility proof
- remaining risk disclosure

Task-specific checks can require screenshots, runtime hook proof, or backend contract handoffs.

## Benchmark Record

Schema: [`schemas/v0/benchmark-record.schema.json`](../schemas/v0/benchmark-record.schema.json)

Benchmark records compare `single_agent`, `crew`, `agent_team`, and `dynamic_workflow` runs by economics and acceptance.

They include rower count, mailbox count, evidence count, blocker count, false done count, wall-clock time, premium token ratio, outcome, and economics verdict.

Native efficiency fields distinguish hard telemetry from soft planning estimates:

- `team_wall_clock_seconds`
- `serial_lower_bound_seconds`
- `hard_telemetry_seconds`
- `rower_self_estimate_seconds`
- `soft_estimate_seconds`
- `confidence_level`

## Workflow Plan

Schema: [`schemas/v0/workflow-plan.schema.json`](../schemas/v0/workflow-plan.schema.json)

Dynamic Workflow Readiness v0 adds a provider-neutral plan IR above normal crew runs.

It records:

- workflow id
- goal
- workspace root
- budget and concurrency limits
- human approval requirement
- ordered phases
- quality patterns such as `map_reduce`, `refuter`, `claim_vote`, `independent_verify`, and `fix_loop`
- approval gates

Default phases are `discover`, `shard`, `fanout`, `cross_check`, `synthesize`, and `verify`.

Related event types:

- `agentic.mode.assessed`
- `agentic.mode.required`
- `agentic.mode.selected`
- `workflow.plan.created`
- `workflow.phase.started`
- `workflow.phase.completed`
- `workflow.agent.spawned`
- `workflow.agent.stopped`
- `workflow.control.requested`
- `workflow.supervision.blocked`
- `workflow.acceptance.completed`
- `claim.submitted`
- `claim.reviewed`

Workflow claims can be gated with `evidence gate --task-type workflow_claim`.

## Mailbox Model

Mailbox messages are represented as events and optionally materialized to per-agent inbox/outbox files.

The event log remains the source of truth.

Message types should map to event payloads:

- `question`
- `contract`
- `status`
- `blocker`
- `review`
- `evidence`

Each message should include:

- sender
- recipient
- related task id
- message type
- body
- delivery state

For v0.1, delivery can be soft real-time. Hooks, MCP tools, or plugins can write events immediately. Agents without push support can poll their inbox file.

## ID Conventions

Use stable, readable prefixes:

- `run_`
- `agent_`
- `task_`
- `msg_`
- `cmd_`
- `evt_`
- `evidence_`

IDs should be unique within the local run store and safe for filenames.

## Design Rule

DragonBoat should never require a worker to claim "done" without evidence.

Evidence is the product boundary between worker execution and steerer trust.
